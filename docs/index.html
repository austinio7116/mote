Overview

Mote is a native C game engine, console OS, and IDE for the Thumby Color — a dual-core RP2350 handheld with a 128×128 RGB565 screen. Games are plain C modules that link no engine code: the resident OS hands each game a jump table of engine functions and drives its frame loop.

This page documents the complete developer-facing surface:

  • The engine API (MoteApi) — the stable ABI of function pointers a game calls through.
  • The data types games construct and pass (meshes, sprites, bodies, splats…).
  • The header-only math and SDK helpers compiled directly into the game (no ABI cost).
  • The module / ABI format and version history.
Hardware target. 128×128 RGB565 display · dual Cortex-M33 @ ~150 MHz · ~280 KB game arena · 9 buttons (D-pad, A, B, LB, RB, MENU) · 22050 Hz audio. The host build (Linux/Windows SDL2 emulator) mirrors the same ABI exactly.

Programming model

A game is loaded by the OS (on the host via dlopen of a .so; on the device by mapping a .mote image and jumping to it). The contract is small and forever-stable:

  1. The OS reads the game's exported ABI version and refuses a mismatch (a game built against a version newer than the engine is rejected).
  2. The OS calls mote_game_register(api); the game stashes the const MoteApi* and returns its MoteGameVtbl.
  3. The OS owns the frame loop and drives the vtable callbacks (init, then update/render_band/overlay each frame).

Everything stateful goes through MoteApi; math and data formats are header-only and compiled into the game directly.

Immediate mode. There is no retained scene graph. Each frame, update() starts the scene and re-submits everything to draw; the draw lists are cleared for you next frame. Keep your own state and draw from it. It holds 60 fps at thousands of triangles, so re-submitting the whole scene every frame is the intended pattern.
ABI rule. MoteApi and MoteGameVtbl are only ever appended to — never reordered or removed — and MOTE_ABI_VERSION is bumped on any change. A game built against an older version runs unchanged on a newer engine; new fields it never calls stay harmless.
C standard library. Game code may use the standard C library: string/memory (memcpy, memmove, strcmp…), math (sinf, sqrtf… — -lm is linked), and the printf family for formatting (snprintf into your own buffer, then draw it with text()). The device build links tiny libc syscall stubs (sdk/mote_syscalls.c) so stdio's formatting code resolves; a game does no real file I/O, so the stubs are never called. Do not use malloc/free — the libc heap isn't wired up; use mote->alloc() (load-time) and file-scope statics instead.

Game skeleton

/* mygame/src/game.c */
#include "mote_api.h"
#include "mote_build.h"      // header-only helpers + auto-includes your baked launcher icon

MOTE_GAME_MODULE();             // stashes `mote`, exports the abi-version symbol

#ifdef MOTE_MODULE_BUILD       // device build only
#include "mote_module.h"
MOTE_MODULE_HEADER();
#endif

static const Mesh *cube;

static void g_init(void) {
    mote->scene_set_background(MOTE_RGB565(10,12,26));
    mote->scene_set_sun(v3_norm(v3(0.4f,0.7f,-0.6f)));
    cube = mote_mesh_box(mote, 1,1,1, MOTE_RGB565(120,180,230));
}

static void g_update(float dt) {
    const MoteInput *in = mote->input();
    Mat3 cam = mote_camera_look(v3(0,0,0), v3(0,0,1));
    mote->scene_camera(&cam, v3(0,0,-5), 60.0f);
    mote_draw(mote, cube, v3(0,0,0));
}

static const MoteGameVtbl k_vtbl = {
    .init = g_init, .update = g_update,
    .config = { .max_tris = 256, .depth = 1 },
};
static const MoteGameVtbl *mote_game_vtbl(void) { return &k_vtbl; }

Build / run / push

mote new mygame # scaffold (use -t 3d|physics|2d for a starter template)
mote run mygame # build host .so + launch the SDL emulator
mote push mygame --launch # cross-build the .mote, upload over USB, run it
mote studio # the IDE (asset authoring + live emulator)

Lifecycle & the vtable

The game returns a MoteGameVtbl of callbacks the OS drives.

CallbackWhenNotes
initonce, after registerBuild/load meshes, sounds, terrain; allocate from the arena.
update(dt)every frame (core 0)Read input, step state, submit the scene. dt in seconds.
render_band(fb, y0, y1)every frame, both coresOptional custom raster (raycaster games). NULL ⇒ the built-in 3D scene rasteriser. Disjoint row bands.
overlay(fb)every frame (core 0)Optional HUD drawn straight into the framebuffer after raster.
configread before initResource pools (see below). Read by the OS to size the arena.

The MENU button's 3-second solo long-hold is reserved by the OS (opens the engine menu / “return to lobby”); all other input is the game's.

Memory model & MoteConfig

The game declares the resource pools it needs in config. At load the OS sizes the engine's buffers to exactly that (not a fixed worst case), allocates them from one shared SRAM arena, and hands the remainder to the game through alloc(). A pool left 0 costs nothing and opts that subsystem out entirely. If the declared pools plus the game's allocations don't fit, the loader refuses to launch (a clear failure, not a crash).

FieldTypeMeaning
max_trisuint163D triangle draw-list capacity. 0 = no 3D raster. Use a baked model's <name>_TRIS.
max_spheresuint16Sphere impostors in the 3D pass.
max_splatsuint16Gaussian splats. 0 = none.
max_spritesuint162D sprites per frame. 0 = no 2D scene.
max_bodiesuint16Physics bodies. 0 = no physics engine.
max_contactsuint16Physics contact manifolds.
max_mesh_trisuint16Largest mesh collider's triangle count. 0 = none.
depthuint81 allocates the 32 KB depth buffer (needed for 3D / splats).
Sizing. A depth=1 game has ~248 KB of arena for the draw-list → roughly 7,000 triangles maximum; framerate gives out before memory does for most games, so size the pool to what you actually draw. The new-game wizard (Studio / mote new -t) fills these in for you.

Engine API — Input v1

All MoteApi calls are made through the mote pointer (e.g. mote->input()).

input

const MoteInput *(*input)(void);

Returns the current frame's derived button state (edges, held timers). Valid during update(). Read it with the helpers mote_pressed / mote_just_pressed / mote_just_released rather than poking the struct. See Input state.

Engine API — 3D scene

Build the frame in update(); the OS rasterises the triangle scene across both cores. Submit world-space geometry once per frame.

scene_set_background v1

void (*scene_set_background)(uint16_t rgb565);

Set the clear colour for the frame.

scene_set_sun v1

void (*scene_set_sun)(Vec3 dir_toward_light_world);

Set the directional light (a world-space unit vector pointing toward the light). Usually set once in init().

scene_camera v13

void (*scene_camera)(const Mat3 *cam_basis, Vec3 cam_pos, float fov_deg);

Begin the 3D scene with a world-space camera. You then add objects at absolute world positions; the engine subtracts cam_pos for you. This is the preferred entry point — pair it with the mote_draw helpers and pass the same cam_pos to scene_set_splats for a consistent scene.

ParameterDescription
cam_basisOrientation basis (rows = right / up / forward). Build with mote_camera_look.
cam_posCamera world position.
fov_degVertical field of view in degrees (≈60 typical).

scene_begin v1

void (*scene_begin)(const Mat3 *cam_basis, float fov_deg);

Legacy camera entry: the camera sits at the origin and you add objects with camera-relative positions (subtract the eye yourself). Equivalent to scene_camera with cam_pos = 0.

scene_add_object / scene_add_object_scaled v1

int (*scene_add_object)(const MoteObject *obj);
int (*scene_add_object_scaled)(const MoteObject *obj, float scale);

Submit one MoteObject (a mesh + position + orientation, with an optional per-draw colour override) to the 3D draw list. The scaled form applies a uniform scale. Most games use the mote_draw helpers instead of building MoteObjects by hand. Returns the new draw-list count (or unchanged when full).

scene_add_sphere v4

int (*scene_add_sphere)(Vec3 cam_rel_pos, float radius, uint16_t color);

Add a cheap per-pixel-shaded sphere impostor, depth-tested with the meshes. Position is camera-relative. Great for balls, particles, projectiles. Counts against max_spheres.

scene_tri_count v1

int (*scene_tri_count)(void);

Triangles submitted so far this frame (for budgeting against max_tris).

Engine API — 2D scene & tiles v2

A screen-space 2D scene the OS rasters after the 3D scene (both banded across cores). A game can be pure-2D, pure-3D, or hybrid (3D world + 2D HUD). Build it in update().

scene2d_begin v2

void (*scene2d_begin)(int cam_x, int cam_y);

Begin the 2D scene; (cam_x,cam_y) is the camera offset applied to all sprite/tile positions at raster time.

scene2d_add v2

int (*scene2d_add)(const MoteSprite *spr);

Add a MoteSprite (a frame of an image at a screen position, with layer + flip/rotate flags). Returns 0 if the sprite pool is full. Counts against max_sprites.

scene2d_set_tilemap v2

void (*scene2d_set_tilemap)(const MoteTilemap *map, const MoteTileset *tiles);

Draw a plain index tilemap: a cols×rows grid of byte indices into a tileset atlas (0xFF = empty).

scene2d_set_autotiles v14

void (*scene2d_set_autotiles)(const uint8_t *terrain, int cols, int rows,
                              const struct MoteAutotile *const *tiles, int n);

Render-time autotiling. Hand the engine a logical terrain map (one byte per cell: 0 = empty, 1..n index tiles) plus a MoteAutotile ruleset per terrain; the raster picks each cell's atlas tile from its 8 neighbours. There is no resolved buffer — re-call it for free when the map changes (procgen, destructible terrain). Build rulesets with mote_autotile_template().

scene2d_set_autotile_layers v15

void (*scene2d_set_autotile_layers)(const uint8_t *map, int cols, int rows,
                                    const struct MoteAutotile *const *tiles, int n);

Layered autotiling. map is one byte per cell where each bit is a layer (bit L set ⇒ tiles[L] drawn there); up to 8 independent, overlapping layers, each autotiled against its own bit and drawn bottom-up. The map is typically a const baked level (lives in flash, zero SRAM).

blit v2

void (*blit)(uint16_t *fb, const MoteImage *img, int x, int y,
             int fx, int fy, int fw, int fh, uint8_t flags, int y0, int y1);

Immediate-mode sprite blit straight into the framebuffer (HUD/UI), band-clipped to rows [y0,y1) and colour-keyed. flags are MOTE_SPR_*. Use from overlay().

Engine API — Physics v3

Impulse-based 3D rigid bodies (spheres, boxes, planes, capsules, convex hulls, static triangle meshes) under gravity, with friction, restitution, rotational inertia, sleeping, and a uniform-grid broad phase. The game owns the body array; the engine runs the solver on it. SI units (metres, kg, seconds). See Physics bodies.

phys_world_defaults v3

void (*phys_world_defaults)(MoteWorld *w);

Fill a MoteWorld with sensible defaults (earth gravity, ~unit walled box, lively bounce). Then override fields (gravity, walls, bmin/bmax, substep…) before stepping.

phys_step v3

uint32_t (*phys_step)(MoteWorld *w, MoteBody *bodies, int n, float dt);

Advance the simulation by dt seconds over the game's bodies[0..n). Uses a fixed-substep accumulator, so it is frame-rate independent. Returns an event bitmask — MOTE_PHYS_HIT when an impact occurred this step (for sound/feedback).

phys_raycast v7

int (*phys_raycast)(const MoteWorld *w, const MoteBody *bodies, int n,
                    Vec3 origin, Vec3 dir, float max_dist, int skip, MoteRayHit *hit);

Cast a ray (unit dir) up to max_dist against every body; returns 1 and fills the nearest MoteRayHit, or 0. Pass skip = a body index to ignore (e.g. the shooter), or <0 to test all. For aiming, ground checks, picking, AI.

phys_overlap v7

int (*phys_overlap)(const MoteWorld *w, const MoteBody *bodies, int n,
                    Vec3 center, float radius, int *out, int max);

Fill out[] with up to max indices of bodies overlapping the test sphere; returns the count.

Static colliders, per-body materials, and walls are plain MoteBody/MoteWorld fields — set them directly. The mesh-collider broad-phase grid is built once with the header-only mote_phys_mesh_build_grid().

Engine API — Gaussian splats v8

A 3D Gaussian-splat renderer (a fourth render path beside raster / raycast / 2D). Each MoteSplat is an anisotropic 3D Gaussian; rendering projects, depth-sorts, and alpha-blends soft ellipses into the framebuffer.

scene_set_splats v8

void (*scene_set_splats)(const MoteSplat *splats, int n, int *order,
                         const Mat3 *cam_basis, Vec3 cam_pos, float fov_deg,
                         const uint16_t *depth);

Register a splat cloud to render this frame as a measured, dual-core banded pass after the 3D scene (composites with depth; its cost shows in the perf graph). Call from update(). order is your scratch of ≥ n ints; depth is depth_buffer() (so rastered geometry occludes splats behind it) or NULL. Preferred over splat_render for anything heavy.

splat_render v8

int (*splat_render)(uint16_t *fb, const MoteSplat *splats, int n,
                    const Mat3 *cam_basis, Vec3 cam_pos, float fov_deg,
                    int *order, const uint16_t *depth);

Immediate splat render from overlay() (blends over the rastered scene). Returns the number drawn.

depth_buffer v8

const uint16_t *(*depth_buffer)(void);

The scene depth buffer (K/z, larger = nearer). Pass to the splat calls for correct occlusion.

Engine API — Audio v10

A small polyphonic synth mixed to 22050 Hz (SDL on host, 12-bit PWM on device). Master volume follows the engine menu.

audio_note / audio_off v10

void (*audio_note)(float freq, float amp);
void (*audio_off)(void);

Strike a one-shot note with a piano-ish decay (freq Hz, amp 0..1) — fire one per key press; voices are stolen when all 8 are busy. audio_off() silences everything.

audio_play v12

void (*audio_play)(const MoteSound *snd, float gain);

Fire a one-shot 22050 Hz mono PCM MoteSound (a baked WAV, or a recipe baked with mote_sfx_bake). gain 0..1+. Up to 4 samples mix at once (oldest stolen); they sum on top of the synth.

audio_render_sfx v19

int (*audio_render_sfx)(const MoteSfx *recipe, int16_t *out, int max);

Synthesise a MoteSfx recipe to 22050 Hz mono PCM. Returns the sample count; writes into out only when out != NULL and bounded by max. Call twice (measure, then render into an arena buffer) — exactly what mote_sfx_bake() does — so a game ships tiny recipes instead of WAVs.

Engine API — Text & overlay v5

text / text_2x v5

int (*text)(uint16_t *fb, const char *s, int x, int y, uint16_t color);
int (*text_2x)(uint16_t *fb, const char *s, int x, int y, uint16_t color);

Draw bitmap text into the framebuffer (1× and 2× scale); returns the advanced x. Draw from overlay(). For printf-style formatting use the helper mote_textf.

Engine API — Control, arena & telemetry

micros v1

uint64_t (*micros)(void);

Monotonic microsecond clock (good RNG seed; manual timing).

exit_to_launcher v1

void (*exit_to_launcher)(void);

Return to the OS launcher (ends the game's frame loop).

set_fps_limit v21

void (*set_fps_limit)(int fps);

Cap the frame rate. 0 (default) runs uncapped — the device free-runs, the host emulator runs as fast as the machine allows. A positive value paces the main loop to that many fps on both device and host, so a game can lock 30/60 fps for steady timing. Call once from update() on the first frame (or change it any time); the async LCD flush overlaps the wait, so capping adds no latency.

menu v11

int (*menu)(const char *title, const char *const *items, int n);

A styled modal menu in the system look. Lists n labels, UP/DOWN to move; returns the chosen index (A) or -1 (B / quit). Blocking — call from update() for pause / game-over / level menus.

alloc / arena_free v9

void   *(*alloc)(uint32_t bytes);
uint32_t (*arena_free)(void);

Carve the game's own buffers out of the same arena the engine pools came from. 8-byte aligned and zeroed; returns NULL if exhausted. Valid from init() onward; freed wholesale on exit. arena_free() reports the bytes left for the game. There is no per-allocation free — regenerate game-owned buffers in place rather than re-allocating.

log / perf v6

void (*log)(const char *s);
void (*perf)(uint32_t out[6]);

log() streams a line to the host (mote logs). perf() fills [fps, update_us, raster_us, flush_us, core0_pct, core1_pct] for the latest frame.

Data types — Meshes mote_mesh.h

Meshes live in flash as quantised int8 vertices + 6-byte faces. Winding: counter-clockwise viewed from outside. world_pos = vert × (scale / 127), so an int8 of ±127 spans ±scale.

MeshVert

typedef struct { int8_t x, y, z; } MeshVert;

MeshFace

typedef struct { uint8_t a, b, c; int8_t nx, ny, nz; } MeshFace;  /* 6 bytes */

a,b,c are vertex indices (uint8 ⇒ a single mesh caps at 256 vertices); nx,ny,nz are the quantised model-space face normal (unit×127). Colour is not per-face — it lives on the Mesh.

MoteHull

A convex hull in local space for MOTE_SHAPE_HULL: verts drive the support function; fnorm + faceoff/facev + edges give SAT axes and face clipping for stable multi-point manifolds. Build with the obj2hull baker or by hand.

MoteMesh

A static concave triangle-mesh collider for MOTE_SHAPE_MESH (verts + uint16 tris, no vertex cap). Add an optional XZ broad-phase grid with the header-only builder so a body tests only nearby triangles.

mote_phys_mesh_build_grid header-only

int mote_phys_mesh_build_grid(MoteMesh *m, int grid_n, uint16_t *start, uint16_t *tri, int tri_cap);

Call once after filling verts/tris. Caller owns the scratch (start = grid_n*grid_n+1 uint16, tri ≥ ntris uint16). Returns 0, or −1 on bad args. Pick grid_n so a cell is ~one triangle wide.

Module format & ABI mote_module.h

A built device game is a flat image: a MoteModuleHeader first, then .text/.rodata, then the .data init image. It is linked at a fixed virtual window (MOTE_MODULE_VADDR = 0x10800000); the loader points an RP2350 ATRANS window at wherever it lives in flash, so it executes in place (XIP) with no copy and no relocation. Mutable state lives at the reserved RAM region (MOTE_MODULE_RAM = 0x20060000, 128 KB).

Header fieldTypeDescription
magicuint32_tMOTE_MODULE_MAGIC ('MOTE').
abi_versionuint32_tMOTE_ABI_VERSION the module was built for.
regfn ptrEntry: mote_game_register.
data_load / data_start / data_end / bss_start / bss_enduint32_tMini-crt ranges (copy .data, zero .bss).
icon_vaddruint32_tv20 Linked address of mote_game_icon_data[], or 0 for none. The launcher reads it straight from flash without loading the game. Format follows the module's ABI: a raw 60×60 RGB565 array up to v21, a compact paletted blob (sdk/mote_icon.h) from v22.

A game emits it with MOTE_MODULE_HEADER() (device build only, guarded by MOTE_MODULE_BUILD). The icon is picked up via a weak symbol that the build auto-includes — mote_build.h pulls in the baked src/icon.h via __has_include, so a game with an icon.png carries its icon with no #include and no boilerplate. From v22 the baked data is a paletted, adaptive-bit-depth blob (~2–4× smaller than raw RGB565); the launcher decodes it inline when drawing the selection.

ABI version history

Each bump appends to MoteApi/MoteGameVtbl or extends a shared data struct. A game runs on any engine whose version is ≥ the one it was built against. Changing the engine's shared structs needs a reflashed OS firmware; header-only/SDK changes do not.
