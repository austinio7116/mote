# ThumbyCraft → Mote — full port plan

Port the bare-metal RP2350 Minecraft-style voxel game **ThumbyCraft**
(`/home/maustin/thumby-color/ThumbyCraft`) to run as a **Mote game module**
(`examples/thumbycraft/`), keeping **every feature**, and fitting it inside Mote's
memory model.

This is a *faithful vendored port* (same discipline as ThumbyCue / Indemnity):
vendor the engine-agnostic `craft_*.c` gameplay/render/gen code unchanged, replace
only the bare-metal device/host shell with a thin Mote vtbl shim.

---

## 1. Why it fits — the memory math (the whole ballgame)

Mote's loader gives a game two RAM regions (`os/device/os_memmap.ld:27`, `sdk/game.ld:19`):

| Region | Size | What goes here |
|--------|------|----------------|
| **Module static `.data`/`.bss`** (`GAME_RAM` @ 0x20060000) | **128 KB** | the module's own global arrays |
| **Arena** (`mote->alloc`, `MOTE_ARENA_SIZE` `os/mote_os.c:29`) | **272 KB** | runtime allocations; a `render_band` game uses **none** of the engine 3D pools, so the **whole** arena is free |
| **= usable RAM** | **400 KB** | |
| Flash (XIP, **zero SRAM**) | 4 MB window | all `const` — code, baked textures, audio data |

The framebuffer (32 KB) is **OS-owned** in Mote (`mote_launcher_fb()`), so unlike the
standalone build it does **not** come out of the game's budget.

**ThumbyCraft's RAM need, re-homed:**

| Buffer | Size | Home | Notes |
|--------|------|------|-------|
| World blocks `64³×1B` | **256 KB** | **arena** (`mote->alloc`) | too big for the 128 KB BSS region — must be arena |
| Lightmap (2 bit × 262144) | 64 KB | module BSS | |
| Z-buffer 128×128×1 | 16 KB | module BSS | we keep ThumbyCraft's `uint8` z-buf; set `config.depth = 0` (skip Mote's 32 KB depth) |
| Skyheight 64×64 | 4 KB | module BSS | |
| Biome map 64×64 | 4 KB | module BSS | |
| Animated tex frames (water/lava/portal) | ~4 KB | module BSS | |
| Player / inventory / hotbar | ~1 KB | module BSS | |
| Mobs (state + sprites) | ~8 KB | module BSS | |
| Redstone / torch state | ~4 KB | module BSS | |
| Menu / HUD / save scratch | ~4 KB | module BSS | |
| Coarse raycast grid 16×16 + sky LUT | ~0.5 KB | arena | |

**Totals:** arena ≈ **257 KB / 272 KB** (≈15 KB free for SFX PCM); module BSS ≈
**109 KB / 128 KB** (≈19 KB free). **Comfortable — no feature needs cutting.**

**Flash (const, zero SRAM):** code `.text` ≈150 KB, baked textures ≈157 KB, audio
data ≈96 KB — all land in the module's 4 MB XIP flash window.

> **Hard constraint that drives the design:** the 256 KB world array **cannot** be a
> static global (module BSS is only 128 KB). It must be `mote->alloc()`'d in `init()`
> and threaded as a pointer. Everything else fits comfortably in static BSS.

## 2. The render/tick mapping is 1:1 (ThumbyCraft was practically built for this)

ThumbyCraft already factors its frame into the exact shape Mote's vtbl wants, and
**both already do dual-core top/bottom banding**:

| ThumbyCraft (`craft_main.h`) | Mote vtbl (`MoteGameVtbl`) | Core |
|------------------------------|---------------------------|------|
| `craft_main_tick(in, dt)` | `update(dt)` | core 0 |
| `craft_main_render_begin()` | (tail of `update`) — precompute camera basis | core 0 |
| `craft_main_render_strip(y0, y1)` | **`render_band(fb, y0, y1)`** | **both cores, concurrently** |
| `craft_main_draw_hud(fps)` | `overlay(fb)` | core 0 |

Mote calls `render_band` once per band per frame, core 0 on `[0,64)` and core 1 on
`[64,128)` — exactly ThumbyCraft's strip split. **Banding rule:** all state
`render_band` reads must be set in `update()` and treated read-only during the band
(no mid-frame global toggles — see `feedback_dualcore_raster_no_global`). ThumbyCraft
already obeys this (it precomputes the camera basis in `render_begin`).

Input maps directly: Mote buttons (`UP/DOWN/LEFT/RIGHT/A/B/LB/RB/MENU`) are the same
physical set ThumbyCraft uses; the shim fills `CraftInput` and ports all 6 control
schemes verbatim.

## 3. Source layout (vendored, mirrors ThumbyCue)

```
examples/thumbycraft/
  game.toml                  # name/author metadata
  icon.png                   # baked -> src/icon.h
  PLAN.md                    # this file
  src/
    game.c                   # NEW — the Mote vtbl shim (init/update/render_band/overlay)
    craft_*.c / craft_*.h    # VENDORED unchanged: world, gen, blocks, render, player,
                             #   main, mobs, redstone, torches, audio, menu, save, hud, types
    craft_textures_baked.c   # generated by the bake step (const, flash)
    <audio data>.h           # vendored const audio (CDL) — flash
```

The Mote build globs every `src/*.c` (`tools/mote:56`) and links them into one
`.mote` module — no per-file wiring needed (ThumbyCue does exactly this).

**What is dropped:** only the bare-metal shell — `device/` (LCD driver, dual-core
launch, flash MSC) and `host/` (SDL). Mote supplies all of that. **No gameplay,
render, gen, mob, redstone, or audio-synth code changes.**

## 4. The shim: `src/game.c`

```c
#include "mote_api.h"
#include "mote_build.h"
MOTE_GAME_MODULE();
#ifdef MOTE_MODULE_BUILD
  #include "mote_module.h"
  MOTE_MODULE_HEADER();
#endif
static const MoteApi *mote;
#include "craft_main.h"   /* + craft_world.h for the world-pointer hook */

static void g_init(void) {
    /* 256 KB world from the arena; everything else is static BSS in the craft_*.c TUs */
    void *world = mote->alloc(CRAFT_WORLD_VOXELS);     /* 262144 B */
    craft_world_set_buffer(world);                     /* NEW tiny setter (see §6) */
    craft_main_init();                                 /* generates spawn window */
}
static void g_update(float dt) {
    CraftInput in; map_buttons(mote->input(), &in);
    craft_main_tick(&in, dt);
    craft_main_render_begin();
}
static void g_render_band(uint16_t *fb, int y0, int y1) {
    craft_set_framebuffer(fb);                         /* NEW tiny setter; per-band-safe */
    craft_main_render_strip(y0, y1);
}
static void g_overlay(uint16_t *fb) {
    craft_set_framebuffer(fb);
    craft_main_draw_hud(/*fps*/ mote_perf_fps());
}
static const MoteGameVtbl k_vtbl = {
    .init = g_init, .update = g_update,
    .render_band = g_render_band, .overlay = g_overlay,
    .config = { 0 },          /* no engine 3D pools — render_band owns the frame; depth=0 */
};
static const MoteGameVtbl *mote_game_vtbl(void) { return &k_vtbl; }
```

## 5. Audio — the one genuine porting decision

ThumbyCraft owns a complete synth (music + SFX + ducking + altitude pitch-shift) that
emits PCM through a **pull callback**: `craft_audio_render(int16_t *out, int n)`
(`craft_audio.h:32`). Mote's audio ABI is higher-level — `audio_note(freq,amp)`,
`audio_play(MoteSound)`, `audio_render_sfx(recipe)` — with **no raw-PCM-stream hook**.

Re-expressing the synth in Mote primitives would **lose** the procedural music and
ducking — violating "no lost features." So:

**Add ABI v36: a streaming-PCM source hook.** A small append-only addition letting a
game register a pull callback the Mote mixer blends at 22050 Hz:

```c
/* ABI v36 — register a PCM source the engine mixer pulls each audio block.
 * fill(out,n) writes up to n mono int16 @ 22050 Hz; returns samples written.
 * NULL unregisters. Mixed alongside note/play/sfx voices. */
void (*audio_set_stream)(int (*fill)(int16_t *out, int n));
```

The shim then calls `mote->audio_set_stream(craft_audio_render)` in `init()` and the
entire ThumbyCraft audio system works unchanged. This is reusable (future emulator-style
slots want it too — fits the IDE-first "engine gains reusable features" philosophy).

**Must accompany the ABI bump** (`feedback_engine_doc_sync`): bump `MOTE_ABI_VERSION`
to 36, wire it in `os/mote_os.c` + the host platform mixer, and update
`docs/index.html` + `README.md` API ref + `CHANGELOG.md` in the same change.

## 6. Tiny seams to add to the vendored code (minimal, non-invasive)

The standalone build owns its world buffer + framebuffer as globals. The port turns
those two into injectable pointers — the only edits to vendored files:

1. `craft_world`: world array becomes a `static uint8_t *` set by
   `craft_world_set_buffer(void*)` (instead of a 256 KB static). One pointer indirection;
   the hot `IDX` macro is unchanged.
2. `craft_render`/`craft_main`: framebuffer becomes a `static uint16_t *` set per-band by
   `craft_set_framebuffer(fb)` (Mote hands us `fb` each `render_band`; it's stable for the
   whole frame, so this is band-safe). Z-buffer stays a static BSS global.

Everything else (gen, blocks, mobs, redstone, torches, menu, save, hud, audio) compiles
**verbatim**.

## 7. Textures — keep procedural, bake to flash const

ThumbyCraft generates its 315×16×16 RGB565 atlas **procedurally** (perlin-blended
stone/dirt/wood, animated water/lava/portal, biome tinting) and bakes it to a `const`
flash array via `tools/bake_textures.c`. Procedural generation is *the design* here, so
this is the documented procgen exception to the editable-asset rule
(`feedback_examples_real_assets`) — and baking to flash `const` is **required** (the
157 KB atlas can't share the 272 KB arena with the 256 KB world).

Plan: port the bake step so `craft_textures_baked.c` is generated at build time and
linked into the module's flash `.rodata` (zero SRAM). The animated water/lava/portal
frame buffers (~4 KB) stay in module BSS and are refreshed at runtime as today.

## 8. Save/load

ThumbyCraft persists player + inventory + per-chunk edit deltas to flash. Map to Mote's
`mote->save(slot, blob, len)` / `load` / `save_slots()`. Action item: confirm Mote's
per-slot blob size limit accommodates the delta store; if deltas can exceed it, cap/compact
the delta set (the world is procedural, so only edits need storing). Low risk.

## 9. Risk register

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **XIP code latency.** ThumbyCraft puts `trace_ray` in SRAM for zero XIP stall; Mote modules execute `.text` from flash XIP (`game.ld` has no RAM-text section). The 16384-rays/frame hot loop could be slower. | **HIGH — go/no-go** | RP2350 has a 16 KB XIP cache; the hot loop is small and tight, both cores share the cache, and `clk_peri` is already at 280 MHz. **Measure FPS on device in Phase 3.** If short: (a) copy the hot raycaster into an arena RAM buffer at init and call via fn-pointer, or (b) add a small RAM-text region to `game.ld`. |
| 2 | **Audio stream hook** doesn't exist in the ABI. | MED | Add v36 `audio_set_stream` (§5) — small, principled, reusable. |
| 3 | **Module BSS overflows 128 KB.** | LOW | Budget ≈109 KB; verify against the link map in Phase 1. If tight, move biome/skyheight maps into the arena (15 KB spare there). |
| 4 | Save delta store exceeds Mote slot size. | LOW | Compact/cap deltas (§8). |
| 5 | Multi-file module build. | RESOLVED | `mote build` globs `src/*.c`; ThumbyCue proves it. |

## 10. Phasing (de-risk memory then perf, earliest)

- **Phase 0 — Scaffold** (½ day): create the dir, `game.toml`, `src/game.c` with a
  trivial gradient `render_band`. Build host + device, push. Proves the module loads &
  draws.
- **Phase 1 — World + gen + render on HOST** (2–3 days): vendor
  `craft_{types,world,gen,blocks,render}`. Wire the world-pointer (arena) + BSS buffers.
  `render_band → craft_main_render_strip`. **Verify terrain renders on host; verify
  `arena_free()` + link map confirm the budget.** ← memory go/no-go.
- **Phase 2 — Player + interaction** (2 days): vendor `craft_player`. Movement,
  collision, auto-step, ladders, place/break, raycast pick, inventory; map all 6 control
  schemes. Host-playable.
- **Phase 3 — DEVICE bring-up + perf** (1–2 days): build `--device`, push, read
  `mote->perf` FPS. ← **performance go/no-go (Risk 1).** Tune if needed.
- **Phase 4 — Mobs / redstone / torches / lighting** (3 days): vendor
  `craft_{mobs,redstone,torches}`; lightmap propagation; billboarded mob sprites; arrows
  & combat; day/night.
- **Phase 5 — Audio** (2 days): add v36 `audio_set_stream`; vendor `craft_audio`; wire
  `craft_audio_render`; doc-sync.
- **Phase 6 — Menus / save / HUD / crafting** (2 days): vendor `craft_{menu,save,hud}`;
  map saves to `mote->save`; furnace/chest/crafting UI; pause menu.
- **Phase 7 — Texture bake + parity + polish** (2 days): build-time texture bake to
  flash; animated water/lava/portal; biome tinting; full side-by-side parity pass vs the
  original; icon.

**Estimated ~14–17 working days**, gated by the two go/no-go checks (Phase 1 memory,
Phase 3 perf). Phases 0–3 prove the hard parts before the bulk of the feature vendoring.
